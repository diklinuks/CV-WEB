Floating Waste Detection

Computer-vision pipeline for identifying floating waste in surface water from drone footage.

  • Python
  • PyTorch
  • YOLO
  • OpenCV
  • NumPy
Repo currently private

The problem

Surface waste in canals, harbours, and lakes is hard to monitor at scale. Manual inspection of long drone flights doesn't fit anyone's workflow. A vision pipeline that scans footage automatically, flags where waste appears, and produces structured output for reporting solves a real bottleneck for environmental monitoring teams.

What it does

An object-detection model from the YOLO family is trained to identify floating waste — bottles, plastic fragments, larger debris — in frames sampled from drone footage. Each detection is tagged with class, confidence score, and frame timestamp. Detections aggregate across a flight into per-location summaries downstream services can act on.

Pipeline

  1. Sample frames from drone footage at a configurable rate (default: 1 fps)
  2. Preprocess: resize, color normalization, optional histogram equalization for difficult lighting
  3. Run YOLO inference on each frame
  4. Filter by confidence threshold (tunable per deployment)
  5. Aggregate per-frame detections into per-flight summaries with GPS metadata where available
  6. Export to structured JSON / CSV for downstream reporting and remediation routing

Technical choices

YOLO over two-stage detectors (e.g., Faster R-CNN) because drone footage is long, and inference speed matters more than the marginal accuracy gain of a two-stage approach. PyTorch because the YOLOv8 and YOLOv11 reference implementations are PyTorch-native and well-maintained.

Augmentation includes water-reflection and lighting variation, since real footage spans cloudy mornings, harsh midday sun, and overcast afternoons. Edge cases that surface from real data rarely surface from clean training sets.

Where it sits in a larger workflow

The detection pipeline is intentionally narrow — it answers "waste appears in frame N at coordinates X, Y." Geographic aggregation, alerting, and remediation routing are separate services that consume the structured output. Keeping the boundary clear means the model is improvable in isolation from the reporting layer.

What's next

Ongoing work: better performance on low-light and reflective water, faster inference for edge deployment, and an evaluation set drawn from real partner-organisation drone footage rather than purely curated data.
